安全测试思路总结

发表于 2015-09-14   |  

学习安全测试也有一段时间了,零零散散的学习了一些X站姿势,也结合前一阵的测试过程,做一个测试思路的总结.

信息搜集:

1.可以先用Nmap扫扫端口情况,换一换不同扫描策略看看结果.
2.用lijiejie的subdomainbrute.py查查子域名.
3.用dirfuzz.py爆破目录.
4.AppScan探测url备用.
5.可以利用登录或者找回密码等页面爆破网站注册用户.
6.根据域名查看域名注册人信息,IP反查,搜集旁站信息…
7.试试域传送漏洞.

总之就是尽可能的搜集信息,上面是能想到的比较重要的几点.在搜集过程中记录觉得有用的信息,例如:后台管理地址,企业邮箱入口.思路再猥琐一点,也可以搜集客服,HR或者其他一些公司公开人员信息,用于反射型XSS,CSRF,社工等用途.

账户体系检测

1.观察是否存在可爆破的页面,能否突破验证码,IP检测(通过修改X-Forwarded-For)等限制.
2.注册过程中可能出现的漏洞:任意手机注册,短信轰炸,越权绑定存在账户等问题,都需要进行排查.
2.修改密码功能,详情参考密码找回漏洞总结

SQL注入检测

1.将AppScan探测到的url存为txt文件.
2.使用sqlmap -m 参数进行批量扫描.
3.选取可疑url进行手工测试,尝试不同绕过.

XSS检测

1.使用chrome的XSS检测插件进行检测:将检测目标和payload添加至指定位置.
2.选取一些大范围使用以及可能造成存储型XSS的参数进行验证.
3.查看\”>&/等字符的过滤情况.

CSRF检测

1.截获请求,观察其中有无类似防止csrf攻击的参数.
2.修改refer继续请求.
3.如果请求被正常返回,构造csrf攻击页面,通过点击访问进行验证.

越权漏洞检测

1.注册两个账号,分别用chrome和火狐登录.
2.通过替换参数检测越权,通常是 id,uid,username等参数.
3.如果有条件同时检测平行越权和垂直越权.

历史漏洞排查

1.如果产品曾经爆出历史漏洞,可单独列出此项.
2.根据历史漏洞进行排查,验证修复情况.

工具扫描结果

1.AppScan 目前只用过这个,针对扫描结果进行验证,给出分析建议.
2.通常是一些敏感信息泄露:电话,邮箱,内网IP等
3.使用HTTPS协议,密码不能自动补全,httponly限制js操作cookie.

测试总结

1.列举测试过程中发现的问题,并简述修复建议.
2.简要评估网站安全.

按大写的 F 或 O 键，然后按 a-z 可以将进程按照相应的列进行排序。而大写的 R 键可以将当前的排序倒转。

Vim练级攻略

发表于 2015-09-02   |  

vim的学习曲线相当的大（参看各种文本编辑器的学习曲线），所以，如果你一开始看到的是一大堆VIM的命令分类，你一定会对这个编辑器失去兴趣的。下面的文章翻译自《Learn Vim Progressively》，我觉得这是给新手最好的VIM的升级教程了，没有列举所有的命令，只是列举了那些最有用的命令。非常不错。

——————————正文开始——————————

你想以最快的速度学习人类史上最好的文本编辑器VIM吗？你先得懂得如何在VIM幸存下来，然后一点一点地学习各种戏法。

Vim the Six Billion Dollar editor

Better, Stronger, Faster.

学习 vim 并且其会成为你最后一个使用的文本编辑器。没有比这个更好的文本编辑器了，非常地难学，但是却不可思议地好用。

我建议下面这四个步骤：

1.存活
2.感觉良好
3.觉得更好，更强，更快
4.使用VIM的超能力
5.当你走完这篇文章，你会成为一个vim的 superstar。

在开始学习以前，我需要给你一些警告：

学习vim在开始时是痛苦的。
需要时间
需要不断地练习，就像你学习一个乐器一样。
不要期望你能在3天内把vim练得比别的编辑器更有效率。
事实上，你需要2周时间的苦练，而不是3天。

第一级 – 存活

安装 vim
启动 vim
什么也别干！请先阅读
当你安装好一个编辑器后，你一定会想在其中输入点什么东西，然后看看这个编辑器是什么样子。但vim不是这样的，请按照下面的命令操作：

启 动Vim后，vim在 Normal 模式下。
让我们进入 Insert 模式，请按下键 i 。(陈皓注：你会看到vim左下角有一个–insert–字样，表示，你可以以插入的方式输入了）
此时，你可以输入文本了，就像你用“记事本”一样。
如果你想返回 Normal 模式，请按 ESC 键。
现在，你知道如何在 Insert 和 Normal 模式下切换了。下面是一些命令，可以让你在 Normal 模式下幸存下来：

i → Insert 模式，按 ESC 回到 Normal 模式.
x → 删当前光标所在的一个字符。
:wq → 存盘 + 退出 (:w 存盘, :q 退出) （陈皓注：:w 后可以跟文件名）
dd → 删除当前行，并把删除的行存到剪贴板里
p → 粘贴剪贴板
推荐:

hjkl (强例推荐使用其移动光标，但不必需) →你也可以使用光标键 (←↓↑→). 注: j 就像下箭头。
:help → 显示相关命令的帮助。你也可以就输入 :help 而不跟命令。（陈皓注：退出帮助需要输入:q）
你能在vim幸存下来只需要上述的那5个命令，你就可以编辑文本了，你一定要把这些命令练成一种下意识的状态。于是你就可以开始进阶到第二级了。

当是，在你进入第二级时，需要再说一下 Normal 模式。在一般的编辑器下，当你需要copy一段文字的时候，你需要使用 Ctrl 键，比如：Ctrl-C。也就是说，Ctrl键就好像功能键一样，当你按下了功能键Ctrl后，C就不在是C了，而且就是一个命令或是一个快键键了，在VIM的Normal模式下，所有的键就是功能键了。这个你需要知道。

标记:

下面的文字中，如果是 Ctrl-λ我会写成 .
以 : 开始的命令你需要输入 回车，例如 — 如果我写成 :q 也就是说你要输入 :q.

Undo/Redo

→ redo

第四级 – Vim 超能力

你只需要掌握前面的命令，你就可以很舒服的使用VIM了。但是，现在，我们向你介绍的是VIM杀手级的功能。下面这些功能是我只用vim的原因。

在当前行上移动光标: 0 ^ $ f F t T , ;
0 → 到行头
^ → 到本行的第一个非blank字符
$ → 到行尾
g_ → 到本行最后一个不是blank字符的位置。

心得：waf存在于一个完整请求的各个部分，云waf，硬waf，软waf。现阶段可以理解为：我们请求的所有数据，都经过waf，由waf进行识别是否为合法请求。bypass的姿势就是要利用不同场景的特性构造一个不能被waf识别的有效请求，这个请求可能起作用于Web server\web程序\数据库等不同的地方。但是万变不离其中，本质就是寻找waf环境和系统环境间相同语句的不同解析结果

Web Server层的bypass

1.IIS服务器

  • % 特性
    场景：iis+asp
    说明：字符串se%lect在waf层解析出来是se%lect在asp+iis环境中是select

  • %u 特性
    场景：iis+asp,iis+aspx
    说明：iis支出unicode编码将关键字的某个字符以%u006c的格式输入在waf层会通过，在iis环境中解析为正常sql语句从而bypass。

  • 多个unicode码对应同一个字符
    场景：iss+asp
    说明：select中的e对应的unicode为%u0065，但是%u00f0同样会被转换成为e。

    s%u0065lect->select
    s%u00f0lect->select
    疑问:如何知道哪些unicode编码会被解码成相同的字符
    可以参考：wooyun知识库wafbypass-七个姿势

2.apache服务器

  • method绕过
    场景：某些版本的Apache
    说明：发生GET请求时，无论method取任何值，都返回200.
    具体参考
    安全宝SQL注入规则绕过

  • php和Apache畸形boundary绕过

    说明：Php在解析multipart-data的时候有自己的特性，对于boundary的识别，只取了逗号前面的内容，例如我们设置的boundary为—-aaaa,123456，php解析的时候只识别了—-aaaa,后面的内容均没有识别。然而其他的如WAF在做解析的时候，有可能获取的是整个字符串，此时可能就会出现BYPASS。

字符编码详解

发表于 2015-08-19   |  

基本概念

  • 字符是各种文字和符号的总称.

  • 字符集是字符的集合:，不同字符集包含的字符个数不同。常见字符集有：ASCII字符集、ISO8859字符集、GB2312字符集、BIG5字符集、GB18030字符集、Unicode字符集等.

使用哪些字符。也就是说哪些汉字，字母和符号会被收入标准中。所包含“字符”的集合就叫做“字符集”。规定每个“字符”分别用一个字节还是多个字节存储，用哪些字节来存储，这个规定就叫做“编码”。

各个国家和地区在制定编码标准的时候，“字符的集合”和“编码”一般都是同时制定的。因此，平常我们所说的“字符集”，比如：GB2312, GBK, JIS 等，除了有“字符的集合”这层含义外，同时也包含了“编码”的含义。

  • 编码：所谓编码就是用数字表示字符，例如用D7D6表示“字”。编码是将字符转化为数字的约定，规则。

  • UNICODE(UCS,Unicode Character Set)字符集为各种语言的每一个字符设定了统一并且唯一的数字编号，其他的编码方式通常是针对某一区域的语言，文字设计，只支持特定的语言。

  • 代码页：规定了适用于特定区域的字符集，以及编码方式。可以理解为字符和字节数据的映射表。不同代码页的编号可以理解为内码，如936为简体中文GBK，950为正体中文Big5。

  • utf(UCS Transformation format):UCS只规定了如何编码，并没有规定这些编码如何保存，传输。于是就有了utf，可以理解为utf是基于UCS的编码方案
    UTF-8就是以8位为单元对UCS进行编码。从UCS-2到UTF-8的编码方式如下：

UCS-2编码(16进制) UTF-8 字节流(二进制)
0000 - 007F 0xxxxxxx
0080 - 07FF 110xxxxx 10xxxxxx
0800 - FFFF 1110xxxx 10xxxxxx 10xxxxxx

例如“汉”字的Unicode编码是6C49。6C49在0800-FFFF之间，所以肯定要用3字节模板了：1110xxxx 10xxxxxx 10xxxxxx。将6C49写成二进制是：0110 110001 001001， 用这个比特流依次代替模板中的x，得到：11100110 10110001 10001001，即E6 B1 89。

总结

UCS为所有的语言的所有字符分配了统一并唯一的数字，UCS结合代码页对现在已经流行起来的一些编码方式做出支持。
文字实际上是以上述表格中utf-8的形式保存在系统中。当我们打开这个文件时(以utf-8为例)：
1.文字首先以utf-8的编码格式保存在文件中。
2.Windows在读取文件时通过BOM确认编码方式为utf-8，以选择对应的代码页65001。
3.按照编码规则 utf-8 -> unicode
4.Windows按照Unicode在指定代码页中查找字体图像，画到窗口上。

安全测评反馈记录

发表于 2015-08-18   |  

SQL注入

等级：紧急

描述：SQL注入利用现有应用程序，将恶意的SQL命令注入到后台数据库引擎执行，它可以通过在Web表单构造恶意SQL语句得到一个存在安全漏洞的网站上的数据库，而不是按照设计者意图去执行SQL语句.
可以直接或间接的获取数据库内容，危害较大.

建议：

1.过滤客户端提交的敏感字符，提交的区域包括：地址栏中的所有参数，请求头信息，cookie中的参数，需要过滤的字符如下：
|–&–;–$–%–@–';
"–<–>–(;)–+–CR;
LF–,–.–select–from;
update–delete–union–and–or;
2.白名单策略：定制允许客户端输入的长度，类型.
3.做好数据库用户权限管理，只给用户开放必要权限.

XSS跨站脚本攻击

1.反射型xss
等级：低

描述：恶意攻击者往Web页面里插入恶意html代码，当用户浏览该页之时，嵌入其中Web里面的html代码会被执行，从而达到恶意攻击用户的特殊目的.
反射型XSS通过给别人发送带有恶意脚本代码参数的URL，当URL地址被打开时，特有的恶意代码参数被HTML解析、执行.
它的特点是非持久化，必须用户点击带有特定参数的链接才能引起.

2.存储型XSS
等级：中

描述：恶意攻击者往Web页面里插入恶意html代码，当用户浏览该页之时，嵌入其中Web里面的html代码会被执行，从而达到恶意攻击用户的特殊目的.
存储型XSS通过上传非法代码，当用户触发恶意代码，浏览器就执行攻击者定义的脚本程序.

建议：
1.对用户输入进行数据合法性验证:

不可信数据将被放置的地方 例子 应该采取的编码 编码格式
HTML标签之间 <div> 不可信数据 </div> HTML Entity编码 &–>&amp;<–>&lt;>–>&gt;”–>&quot;‘–>&#x27;/–>&#x2f;
HTML标签的属性里 <input type=”text”value=” 不可信数据 ” /> HTML Attribute编码 &#xHH;
JavaScript标签里 <script> var msg = ” 不可信数据 ” </script> JavaScript编码 \xHH
HTML页面的URL里 <a href=”/page?p= 不可信数据 ” >…</a> URL编码 %HH
CSS里 div style=” width: 不可信数据 ” > … </div> CSS编码 \HH

2.为Cookie加上HttpOnly标记:
许多XSS攻击的目标就是窃取用户Cookie.
窃取Cookie一般都会依赖JavaScript读取 Cookie信息，而HttpOnly标记的cookie不允许被JS操作，这样即使Web应用产生了XSS漏洞，Cookie信息也能得到较好的保护.

CSRF跨站请求伪造

等级：中

描述：攻击者盗用用户身份，以用户名义发送恶意请求.
CSRF可能造成的危害包括：以用户名义发送邮件，发消息，盗取账号，甚至于购买商品，虚拟货币转账，个人隐私泄露以及财产安全.

建议：
1.验证HTTP Referer字段：
验证每一个请求的Referer值，如果来自本站，是合法的.
如果Referer是其他网站的话，就有可能是CSRF攻击，则拒绝该请求.

2.在HTTP请求中以参数的形式(或是通过XMLHttpRequest这个类加到请求头中)加入一个随机产生的token，并在服务器端建立一个拦截器来验证这个token，如果请求中没有token或者token内容不正确，则认为可能是CSRF攻击而拒绝该请求.

clickjaking点击劫持

级别：低

描述：点击劫持是一种视觉上的欺骗手段.
攻击者使用一个透明的、不可见的iframe，覆盖在一个网页上，然后诱使用户在该网页上进行操作，此时用户将在不知情 的情况下点击透明的iframe页面.
通过调整iframe页面的位置，可以诱使用户恰好点击在iframe页面的一些功能性按钮上.

建议：在头信息中添加X-FRAME-OPTIONS字段，禁止网站使用ifame载入网页.
X-Frame-Options响应头.
赋值有如下三种：

1.DENY：不能被嵌入到任何iframe或者frame中.

2.SAMEORIGIN:页面只能被本站页面嵌入到iframe或者frame中.

3.ALLOW-FROM uri：只能被嵌入到指定域名的框架中.

apache可配置http.conf如下：

<IfModule headers_module>
Header always append X-Frame-Options "DENY"
</IfModule>

页面敏感信息泄露

级别：视泄露信息而定

描述：
各种不同方式的敏感信息泄露给攻击者提供了信息.

1.直接访问敏感信息文件
2.敏感信息(电话号码，内网IP，邮箱)显示在了页面，源码，url地址中.

3.返回页面暴漏敏感信息(服务器版本，服务器物理路径…).

4….发现再补

建议(一一对应):
1.限制对该文件的访问.

2.确认页面中电话号码，IP用途，站点可根据网站自身情况是否去除号码.
3.如果是正常页面中包含路径信息，如果不需要该路径信息，删除该信息.

如果是错误信息中包含路径信息，需要屏蔽应用程序错误信息.

Apache HTTP Server “httpolny”cookie 泄露

1.为了防止XSS，我们通常采用给cookie附上httponly属性，来禁止js获取cookie.

2.原理：当Apache接收到了cookie长度超过4k，返回400，并且将所有的cookie信息返回，提示cookie过长.

3.利用：为用户的cookie添加一些无用信息，总长超过4k，使Apache返回400，并用过js正则过滤掉无用部分，我们就绕过了httponly的设置.

4.修复：受影响的版本为2.21及其之前的所有版本，做好备份，升级Apache.

//将如上恶意代码插入页面，一个正常用户通过这个被劫持的页面进行访问，即使是设置属性httponly的cookie也通过TRACE方法的请求，显示在返回页面中.

4.修复：属于服务器配置不当造成的安全问题，TRACE方法，如非必须请关闭.

IIS短文件名泄露

1.如果网站是运行在IIS服务器上，可以通过”~”来发现一些文件和文件夹,攻击者可以发现重要的文件或者文件夹，如果这些文件或者文件夹是正规的可见文件.攻击者可以通过该途径搜集文件名称(字符长度不超过6).
2.原理：使用通配符*和?发送一个请求到iis,当IIS接收到一个文件路径中包含”~”的请求时，它的反应是不同的.基于这个特点，我们可以根据http的响应区分一个可用或者不可用的文件.
3.修复：

  • 过滤~字符.

  • Windows中禁用8.3格式文件名方法如下(此修改只能禁止NTFS8.3格式文件名创建,已经存在的文件的短文件名无法移除)：

    cmd> fsutil 8dot3name query c:
    “c:”表示查询C盘的状态.
    如果功能被启用，会显示类似如下的信息：

    Disable8dot3 的卷状态为 0 (8dot3 名称创建已启用).
    NtfsDisable8dot3NameCreation 的注册表状态为默认值 2 (卷级别设置).
    基于以上的两个设置，在 c: 上启用 8dot3 名称创建.

如果在C盘上关闭这个功能，则可使用以下命令：

cmd> fsutil 8dot3name set C: 1
//0（全部启动），1（全部禁用），2（每个盘符单独设置），3（除系统盘外全部禁用）.

也可以通过修改注册表实现：

键：HKLM\SYSTEM\CurrentControlSet\Control\FileSystem
值：NTFSDisable8dot3NameCreation  类型：REG_DWORD
  • 使用waf进行防护.

会话标识别未更新

1.会话标识别未更新

2.原理：用户的登录页面会有一个session id，登录成功后，应该使用新的session id 而不是保持登录页面的session id， 这样即使别人在网络上截获了你的登录页面的session id，仍然无法伪造客户身份进入系统.
如果使用同样的session id，则可能被截获并且伪造客户身份进入系统.

3.修复：
-始终生成新的会话，供用户成功认证时登录.
-防止用户操纵会话标识.
-请勿接受用户浏览器登录时所提供的会话标识.

敏感信息泄露

1.以电话号码泄露为例

2.原理：敏感信息(电话号码，内网IP)显示在了页面，源码，url地址中.

3.利用：可作为进一步渗透的信息储备

4.修复：此号码网站业务正常，无实际安全影响，站点可根据网站自身情况是否去除号码.

缺少httponly属性

1.会话cookie中缺少HttpOnly属性会导致攻击者可以通过程序(JS脚本、Applet等)获取到用户的cookie信息，造成用户cookie信息泄露，增加攻击者的跨站脚本攻击威胁.

2.修复：
向所有会话cookie中添加“HttpOnly”属性.

Java示例：
HttpServletResponse response2 = (HttpServletResponse)response;
response2.setHeader( "Set-Cookie", "name=value; HttpOnly"); 

C#示例：
HttpCookie myCookie = new HttpCookie("myCookie"); 
myCookie.HttpOnly = true; 
Response.AppendCookie(myCookie);

VB.NET示例：
Dim myCookie As HttpCookie = new HttpCookie("myCookie") 
myCookie.HttpOnly = True 
Response.AppendCookie(myCookie)

启用不安全的HTTP方法

1.检测到目标Web服务器配置成允许下列其中一个（或多个）HTTP 方法：DELETE, SEARCH,COPY,MOVE, PROPFIND, PROPPATCH, MKCOL ,LOCK ,UNLOCK.这些方法表示可能在服务器上使用了 WebDAV.由于dav方法允许客户端操纵服务器上的文件，如果没有合理配置dav，有可能允许未授权的用户对其进行利用，修改服务器上的文件.
2.利用：
参考：

IIS WebDav Options 方法未禁用

3.修复：
如果服务器不需要支持WebDAV，请禁用.
或者为允许webdav的目录配置严格的访问权限，如认证方法，认证需要的用户名，密码.

目标web应用表单密码类型输入启用了自动完成操作

1.在web应用form表单中，如果input标签没有指定“autocomplete”属性为“off”，则“autocomplete”的属性会自动默认为“on”.
当web应用form表单中的密码类型的input标签的“autocomplete”属性为“on”时，用户若提交了一个新的用户名和密码时，浏览器将会询问用户是否保存该用户名和密码信息.
如果用户选择保存，则之后在显示该web应用表单时，用户名和密码将会被自动填充到对应的输入框中.
用户一但保存密码，攻击者就可以通过本地从浏览器缓存中获取明文密码，导致用户敏感信息泄露.

2.修复：
修改web应用form表单中密码类型input标签的“autocomplete”属性为“off”.

示例：

<input type="password" name="password">
则增加“autocomplete”属性为“off”即可，修改为：
<input type="password" name="password" autocomplete="off">
